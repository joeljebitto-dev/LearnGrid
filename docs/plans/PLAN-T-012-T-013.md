# T-012/T-013 Assessment Authoring And Quiz Attempts Plan

## Summary
Implement T-012 and T-013 primarily in `assessment-service`. Add assessment authoring models/APIs, quiz/exam attempt workflows, durable PostgreSQL answer storage, scoped authorization, service integrations, local event publishing, tests, and docs. Assignment authoring/config is included; assignment submissions stay for T-014.

## Key Changes
- Add `assessment-service` models/migration for canonical assessment tables needed now: `question_banks`, `questions`, `assessments`, `quizzes`, `quiz_questions`, `quiz_attempts`, `quiz_answers`, `assignments`, and `submission_audit_logs`.
- Add settings for `COURSE_SERVICE_BASE_URL`, `USER_SERVICE_BASE_URL`, and `ENROLLMENT_SERVICE_BASE_URL`; mount routes under `/api/assessments/`.
- Implement authoring APIs:
  - `GET/POST /api/assessments/question-banks/`
  - `GET/PATCH/DELETE /api/assessments/question-banks/<uuid>/`
  - `GET/POST /api/assessments/question-banks/<uuid>/questions/`
  - `GET/PATCH/DELETE /api/assessments/questions/<uuid>/`
  - `GET/POST /api/assessments/`
  - `GET/PATCH/DELETE /api/assessments/<uuid>/`
  - `PUT /api/assessments/<uuid>/questions/` for ordered quiz/exam questions
  - `POST /api/assessments/<uuid>/publish/`
  - `POST /api/assessments/<uuid>/close/`
- Implement attempt APIs:
  - `POST /api/assessments/<uuid>/attempts/start/`
  - `GET /api/assessments/attempts/<uuid>/`
  - `PUT /api/assessments/attempts/<uuid>/answers/`
  - `POST /api/assessments/attempts/<uuid>/submit/`
  - `POST /api/assessments/attempts/<uuid>/auto-submit/`
- Validation rules:
  - Supported question types: `multiple_choice`, `multiple_select`, `true_false`, `short_answer`, `essay`, `file_upload`.
  - `coding` remains schema-reserved and returns a controlled validation error.
  - Objective question choices use stable choice IDs; correct answers are hidden from student attempt responses.
  - Quiz/exam start enforces published status, availability window, max attempts, and active enrollment.
  - Randomized question order is persisted in `submission_audit_logs.metadata` at attempt start.
  - Quiz answers are durable in `quiz_answers`; objective scores are computed locally, while full grading remains T-015.
- Authorization:
  - Authoring requires `assessment.manage`, checking course scope first and institution scope after resolving course metadata from course-service.
  - Student reads/attempts require `assessment.view` plus enrollment-service access validation for the current `UserProfile`.
  - Question banks use institution-scoped `assessment.manage/view`.
  - Remote service failures deny or return controlled `502` errors; `/health/` remains public.
- Events:
  - Add local publisher abstraction emitting `AssessmentPublished`, `AssessmentClosed`, `QuizAttemptStarted`, and `QuizSubmitted`.
  - `QuizSubmitted` includes submitted vs auto-submitted status; real Kafka transport remains T-020.

## Assumptions
- No frontend work is required for T-012/T-013.
- Assignment submissions are excluded and remain T-014.
- Full grading records/certificate behavior remain T-015/T-016.
- Redis is not required for this baseline because submitted answers and attempt metadata are durably stored in PostgreSQL.
