# T-018/T-019 Search, Reporting, Analytics, And API Gateway Plan

## Summary
Implement `T-018` in `analytics-service` and `T-019` with an Nginx API Gateway. Use the existing PostgreSQL `analytics_db` tables for T-018 and keep `OD-005 Analytics Storage` open. Resolve `OD-001 API Gateway Selection` to **Nginx**.

## Key Changes
- Extend `analytics-service` using the existing T-011 models: `event_facts`, `dashboard_aggregates`, `report_snapshots`, `usage_metrics`, and `search_index_records`.
- Add a PostgreSQL-only migration for the documented `gin_search_index_search_text` full-text index, with SQLite/test-safe no-op behavior.
- Add analytics APIs:
  - `GET /api/analytics/search/`
  - `GET /api/analytics/search/courses/`
  - `GET /api/analytics/search/users/`
  - `GET /api/analytics/search/enrollments/`
  - `GET /api/analytics/search/assessments/`
  - `GET /api/analytics/search/submissions/`
  - `POST /api/analytics/search/index-records/`
  - `DELETE /api/analytics/search/index-records/<resource_type>/<uuid>/`
  - `GET/POST /api/analytics/dashboards/aggregates/`
  - `GET/POST /api/analytics/usage-metrics/`
  - `POST /api/analytics/reports/generate/`
- Search filters: `q`, `institution_id`, `resource_type`, `status`, `course_id`, `profile_type`, `assessment_type`, `submission_status`, `sort`, `page`, and `page_size`.
- Report generation supports `active_users`, `enrollments`, `completion_rates`, `assessment_results`, and `system_usage`, using only `analytics_db` data. Generated reports are saved as `report_snapshots`.
- Enforce scoped authorization through auth-service:
  - Search uses resource-specific view permissions such as `course.view`, `profile.view`, `enrollment.view`, `assessment.view`, and `submission.view`.
  - Reports, usage metrics, aggregate writes, and index writes use `analytics.view` at institution or platform scope.
  - Remote auth failures deny access.

## API Gateway
- Add an `api-gateway` Docker Compose service using Nginx.
- Local gateway URLs:
  - HTTP: `http://127.0.0.1:8080`
  - HTTPS: `https://127.0.0.1:8443`
- Generate a local self-signed TLS certificate through a tracked script; generated cert/key files stay untracked.
- Route:
  - `/` to frontend-service on `5173`
  - `/api/auth/` to `8001`
  - `/api/users/` to `8002`
  - `/api/courses/` to `8003`
  - `/api/content/` to `8004`
  - `/api/enrollments/` to `8005`
  - `/api/progress/` to `8006`
  - `/api/assessments/` to `8007`
  - `/api/grading/` to `8008`
  - `/api/grades/` as an alias rewritten to `/api/grading/`
  - `/api/notifications/` to `8009`
  - `/api/analytics/` to `8010`
  - `/api/v1/...` rewritten to the current `/api/...` paths
- Configure Nginx TLS termination, JSON access logs, request IDs, CORS headers, rate limits, and request-size limits.
- Update `pnpm dev` and `pnpm dev:fast` to start the gateway after app services are running; keep direct service ports available for debugging.

## Test Plan
- Analytics-service:
  - Search scope enforcement and resource-type filtering.
  - Index upsert/delete behavior.
  - PostgreSQL search fallback behavior for SQLite tests.
  - Event ingestion idempotency remains intact.
  - Dashboard aggregate and usage metric create/list.
  - Report generation for all five report types.
  - No cross-service transactional database joins.
  - Remote authorization denial/network failure denies access.
- Gateway:
  - `docker compose config` validates.
  - Nginx config syntax validates.
  - Gateway health and route smoke tests cover all service prefixes.
  - `/api/v1/...` rewrite works.
  - Oversized requests return `413`.
  - CORS preflight responses are correct.
  - HTTP redirects to HTTPS locally.
- Verification commands:
  - `analytics-service`: `poetry run ruff check .`, `poetry run python manage.py check`, `poetry run python manage.py makemigrations --check --dry-run`, `poetry run pytest`
  - Gateway/dev runner: `bash -n scripts/run-dev.sh`, `docker compose config`, gateway smoke pytest suite
  - Frontend only if Vite proxy/env config changes: `pnpm -C frontend lint`, `pnpm -C frontend typecheck`, `pnpm -C frontend test`, `pnpm -C frontend build`

## Assumptions
- `OD-001` is resolved to Nginx.
- `OD-005` remains open; PostgreSQL `analytics_db` is the current implementation store, not the final long-term analytics-storage decision.
- T-019 is a local/dev gateway baseline, not a production Kubernetes ingress or cloud load-balancer setup.
- No new auth-service RBAC permission is added; analytics operations continue using the existing `analytics.view` permission.
