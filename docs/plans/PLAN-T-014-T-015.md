# T-014/T-015 Assignment Submissions And Grading Plan

## Summary
Implement `T-014` in `assessment-service` and `T-015` in `grading-service`. Assignment submissions will store text and content asset UUID attachments in `assessment_db`; grading-service will own grade rules, grade records, manual reviews, grade history, publication, and published results in `grading_db`. Notification delivery remains future `T-017`; `GradePublished` events are the handoff.

## Key Changes
- Extend `assessment-service`:
  - Add `AssignmentSubmission` model/migration for `DB-ASSESS-009`.
  - Add assignment submission APIs under `/api/assessments/`:
    - `GET/POST /assignments/<uuid>/submissions/`
    - `GET/PATCH /submissions/<uuid>/`
    - `POST /submissions/<uuid>/submit/`
    - `POST /submissions/<uuid>/mark-graded/`
  - Validate published assignment availability, due date, late policy, current student profile, active enrollment, and optional `attachment_asset_id` through content-service.
  - Preserve `submission_audit_logs` for draft/save/submit/late/graded events.
  - Emit local `AssignmentSubmitted` events.

- Add grading-source integration in `assessment-service`:
  - `GET /api/assessments/grading/quiz-attempts/<uuid>/`
  - `GET /api/assessments/grading/assignment-submissions/<uuid>/`
  - These return only grading-safe metadata: submission/attempt ID, student, course, assessment, score where available, max score, status, and feedback/source payload. They require `grade.manage` or equivalent scoped permissions.

- Implement `grading-service`:
  - Add models/migration for `grading_rules`, `grade_records`, `manual_reviews`, `grade_history`, and `published_results`.
  - Leave `certificate_eligibility` and `certificates` for T-016.
  - Add settings for `ASSESSMENT_SERVICE_BASE_URL`, `COURSE_SERVICE_BASE_URL`, and `USER_SERVICE_BASE_URL`.
  - Add scoped `grade.view` and `grade.manage` helpers using auth-service authorization checks.

- Add grading APIs under `/api/grading/`:
  - `GET/POST /rules/`, `GET/PATCH /rules/<uuid>/`
  - `GET /records/`, `GET /records/<uuid>/`
  - `POST /records/calculate/` for objective quiz attempts from assessment-service grading source data
  - `POST /records/manual-reviews/` to create pending manual review for subjective quiz or assignment submissions
  - `POST /manual-reviews/<uuid>/complete/` to record reviewer score and feedback
  - `POST /records/<uuid>/override/` with required `change_reason`
  - `POST /records/<uuid>/publish/` to create `published_results`, mark records published, emit `GradePublished`, and call assessment-service `mark-graded` where applicable
  - `GET /results/`, `GET /results/<uuid>/` for published result visibility

## Behavior And Authorization
- Students can create/read/update only their own assignment drafts and submit only their own available assignments.
- Instructors and Teaching Assistants use `submission.view/manage` for assignment submissions and `grade.view/manage` for grading workflows at course/institution scope.
- Late submissions are rejected when `allow_late_submission=false`; accepted late submissions use status `late`.
- Automated quiz grading uses the durable quiz attempt score and max score from assessment-service.
- Manual review creates immutable `grade_history` entries when scores change.
- Overrides require a non-empty reason and always write grade history.
- Published results are visible to the owning student and to scoped grade managers.

## Assumptions
- No frontend changes are required for T-014/T-015.
- Assignment file uploads remain content-service responsibility; assessment-service stores only `attachment_asset_id`.
- Grade publication emits events now; notification delivery is implemented later in T-017.
- Certificate eligibility and certificate issuance remain T-016.
- Kafka transport remains future T-020; events are local structured publisher calls for now.
